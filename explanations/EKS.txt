--> create IAM role on aws to give or (attach) EKS-Bootstrap-Server instance role so i can create EKS cluster from that instance


🛠️ Step 1: Create the Custom IAM Policy
eksctl needs to create VPCs, CloudFormation stacks, IAM roles, EC2 custom instances, and the EKS cluster itself.

Open the IAM Console.

Go to Policies ➔ Create policy.

Switch to the JSON tab and paste the following scoped-down policy (this grants everything eksctl needs without giving away your entire AWS account):

JSON

{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow",
            "Action": [
                "eks:*",
                "ec2:*",
                "iam:*",
                "cloudformation:*",
                "autoscaling:*",
                "elasticloadbalancing:*"
            ],
            "Resource": "*"
        }
    ]
}

Click Next, name it EKS-Bootstrap-Policy, and click Create policy.


👥 Step 2: Create the IAM Role for EC2
Next, you need to wrap that policy into a role that an EC2 instance can assume.

In the IAM Console, go to Roles ➔ Create role.

For Trusted entity type, select AWS service.

For Service or use case, select EC2 from the dropdown, then click Next.

In the permissions search box, look for the policy you just created (EKS-Bootstrap-Policy) and check the box next to it.

Click Next.

Name the role (EKS-Bootstrap-Role) and click Create role.


🖥️ Step 3: Attach the Role to your Bootstrap Instance
Now, attach this role to your EKS-Bootstrap-Server instance as an Instance Profile.

Open the EC2 Console and click on Instances.

Select your EKS-Bootstrap-Server instance.

Click on Actions ➔ Security ➔ Modify IAM role.

Select EKS-Bootstrap-Role from the dropdown list.

Click Update IAM role. 


--> Note: The EKS-Bootstrap-Policy (which we created)
The JSON policy we wrote defined exactly the services that the eksctl tool needs to build the cluster, which are:

1- EKS: To create the cluster itself.

2- EC2 & AutoScaling: To run the Worker Nodes (the node servers).

3- VPC / Network: To create the internal network and subnets.

4- CloudFormation: Because the eksctl tool uses it in the background to build the infrastructure.

5- IAM: To create the specific roles for the nodes and the cluster.

--> Note: giving instance administrative role is huge security risk because if your instance compromised the attacker will exploit this vulnerability to control and know your aws infrastructure and use it to destroy your infrastructure or know everything you have built, deployed, managed and can change your infrastructure or your cluster configurations

- ArgoCD
---------

1 ) First, create a namespace
    $ kubectl create namespace argocd
    $ kubectl create namespace application    --> we will use this namespace for application deployment

2 ) Next, let's apply the yaml configuration files for ArgoCd
    $ kubectl apply -n argocd -f https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/install.yaml

3 ) Now we can view the pods created in the ArgoCD namespace.
    $ kubectl get pods -n argocd

4 ) To interact with the API Server we need to deploy the CLI:
    $ curl --silent --location -o /usr/local/bin/argocd https://github.com/argoproj/argo-cd/releases/download/v2.4.7/argocd-linux-amd64
    $ chmod +x /usr/local/bin/argocd

5 ) Expose argocd-server
    $ kubectl patch svc argocd-server -n argocd -p '{"spec": {"type": "LoadBalancer"}}'   --> this command creates a CLB which operates at layer 4 (ip address + port)

6 ) Wait about 2 minutes for the LoadBalancer creation
    $ kubectl get svc -n argocd

7 ) Get pasword and decode it.
    $ kubectl get secret argocd-initial-admin-secret -n argocd -o yaml
    $ echo WXVpLUg2LWxoWjRkSHFmSA== (o/p of password in secret file) | base64 --decode   --> to place the decoded password in the argocd UI (should be decoded not encoded)

--> copy the EXTERNAL-IP of argocd-server (DNS) that come from this command ($ kubectl get svc -n argocd) and paste in browser to open argocd then click on proceed to open argocd UI

- name: admin
- password: the o/p of decode command ( $ echo WXVpLUg2LWxoWjRkSHFmSA== (o/p of password in secret file) | base64 --decode)

--> after logging into argocd UI go to user info and press update password to change your password to something you know or memorized well
